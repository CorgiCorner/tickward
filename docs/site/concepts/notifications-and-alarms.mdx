A timer "firing" is a client-side event. When a timer is armed, the hook
schedules its alarm to go off at the exact target time, independent of the
countdown render loop, and keeps a tick-based check as a backstop. Web Push
exists for the one case the client cannot cover: no tab is open at all.

## The local alarm pipeline

The hook in `components/use-local-timer-alarms.ts` owns alarm delivery. A timer
participates only if it opted in - `timer.notify` or `timer.notification.enabled`
is true - and is not archived. Its alarm boundary is the target date, or, for
recurring timers, the next occurrence.

Delivery uses two primary mechanisms, both armed at the absolute target time and
both independent of the render loop, plus a tick-based backstop. All three funnel
through one `fire()` routine guarded by a fired-key set of the form
`timerId::boundary`, so each occurrence fires exactly once no matter which path
reaches it first:

- **Scheduled sound (primary).** `scheduleNotificationSound` in
  `lib/notification-audio.client.ts` starts an `AudioBufferSourceNode` on the Web
  Audio clock at `ctx.currentTime + secondsUntilTarget`. That clock keeps running
  while the tab is hidden, so a pre-scheduled buffer plays at the target even in a
  background tab.
- **Scheduled notification and full-page alarm (primary).** A
  `setTimeout(boundary - now)` calls `fire()` directly. `setTimeout` still fires
  in a hidden tab (throttled by roughly a second), unlike React's commit and
  effect flush, which browsers coalesce while the tab is in the background.
- **Crossing-based check (backstop).** The shared per-second ticker (see
  [Countdown Accuracy](/concepts/countdown-accuracy)) still asks, every tick,
  whether a boundary fell between the previous tick and this one:

  ```ts
  const readyToFire = boundaryMs > prevNowMs && boundaryMs <= nowMs && !firedKeys.has(firedKey)
  ```

  This catches anything the primaries miss - audio that was never unlocked, a tab
  woken after the scheduling horizon, or a boundary already past at mount.

Only boundaries within a ~15-minute horizon are pre-scheduled; longer timers are
armed once they come within range on a later tick, and the backstop covers the
gap. When a timer is added, edited, removed, or its sound changes, the schedule
is cancelled and re-armed; on unmount every scheduled source and timeout is
cleared.

When a timer fires, up to three things happen, each gated by per-device
preferences:

- **Browser notification.** If a service worker controls the page, the hook
  posts a `SHOW_NOTIFICATION` message and `public/sw.js` calls
  `registration.showNotification`; otherwise it falls back to
  `new Notification(...)`. Clicking the notification focuses an existing app
  window or opens a new one. This path requires the per-device toggle and
  `Notification.permission === "granted"`.
- **Sound.** The primary path pre-schedules the chosen sound (`polite`, `glass`,
  `chord`, `alarm`; `none` disables it) on the Web Audio clock, as above. To do
  that the mp3 is decoded into an `AudioBuffer` ahead of time. Browsers block
  audio that was never preceded by a user gesture, so the hook resumes a shared
  `AudioContext` on the first `pointerdown` or `keydown`, decodes the selected
  sound, and starts a near-silent keep-alive source that stops the browser from
  suspending the context while the tab is hidden; it re-resumes on
  `visibilitychange`. If a sound was never decoded in time, `fire()` falls back to
  an HTML `Audio` element and then to a short oscillator tone - the foreground and
  last-resort paths.
- **Full-page alarm.** The hook sets alarm state that the app renders as a
  full-screen takeover, dismissed from the UI.

If none of the three is enabled, the tick is a no-op. The preferences are
per-device, stored in `localStorage` under `notificationsEnabled`,
`timerAlarmFullPage`, and `timerAlarmSound` (`lib/notification-preferences.ts`
defines the keys; `lib/local-notification-preferences.client.ts` keeps React in
sync, including across tabs via the `storage` event). The settings panel in
`components/notification-settings.tsx` calls `Notification.requestPermission()`
when you enable system alerts.

## What works in which tab state

| Tab state | What works |
|-|-|
| Foreground tab | everything: notification, sound, full-page alarm |
| Background tab | everything, as long as audio was unlocked by an earlier gesture (needed only for sound) |
| Browser closed, tab discarded or frozen | nothing local - this is what Web Push is for |

Background tabs work because the alarm is armed at the absolute target time, not
polled by the render loop. The sound rides the Web Audio clock and the
notification and full-page alarm ride a `setTimeout`, both of which keep firing
while the tab is hidden; the per-second crossing check is only a backstop. (An
earlier version relied on that tick alone, which browsers throttle once the tab
loses focus - so a hidden tab's alarm only arrived when you came back to it.)

## Web Push

Web Push is the server-delivered path for when no tab is open. It is optional
and requires VAPID keys - `WEB_PUSH_VAPID_PUBLIC_KEY`,
`WEB_PUSH_VAPID_PRIVATE_KEY`, and `WEB_PUSH_VAPID_SUBJECT` in `.env.example`
(see the [self-hosting guide](/guides/self-hosting)).

The subscription lifecycle lives in `lib/web-push-client.ts`. The client
fetches the VAPID public key from `/api/push/public-key` (the route answers
501 when no keys are configured), subscribes through
`registration.pushManager.subscribe` with `userVisibleOnly: true`, and posts
the resulting subscription - the push endpoint plus its `p256dh` and `auth`
keys - to `/api/push/subscribe` together with the account's restore key. If
the server fails to persist it, the browser subscription is rolled back with
`unsubscribe()`, so the two sides never disagree. Subscriptions are stored
per account behind the repository interface in `lib/web-push-subscriptions.ts`.

Delivery is server-driven. `lib/notification-delivery.ts` defines a
`NotificationDeliveryProvider` with a `sendTimerFinished` command and ships a
null provider that marks every channel as skipped with reason
`provider_not_configured`. The subscription plumbing is always in place, but
no push message is sent until a deployment wires a delivery provider.

## Permissions at a glance

| Mechanism | Notification permission | Push subscription |
|-|-|-|
| Sound, full-page alarm | not needed (sound needs one prior user gesture) | not needed |
| Browser notification (local pipeline) | granted | not needed |
| Web Push | granted | required, plus server VAPID keys |

The sound and full-page alarm exist precisely because they need no permission:
a timer can make itself heard on a device where notifications were never
granted, as long as a tab is open.
