---
title: "Account migration"
description: "Move Tickward projects between Cloud and self-hosted deployments without deleting over-limit data."
---

Tickward account exports use a versioned JSON format. The same file works in
both directions:

- [Self-host to Cloud](#self-host-to-cloud)
- [Cloud to self-host](#cloud-to-self-host)

The export includes project metadata, each complete project snapshot (timers
and spaces), account preferences, and per-target notification rules.

## Self-host to Cloud

1. Sign in to the self-hosted deployment and open **Settings → Account**.
2. Select **Export account** and save the JSON file.
3. Sign in to Tickward Cloud and open **Settings → Account**.
4. Select the JSON file under **Account migration**.

Cloud imports every non-conflicting project. If the account exceeds the Cloud
project limit, the additional projects are imported as read-only instead of
being rejected. The normal over-limit retention notice and cleanup policy then
apply.

## Cloud to self-host

Follow the same steps in reverse: export from Cloud, then import the file in
the self-hosted deployment. The target deployment applies its configured
project limit from the admin-managed plan limits, or its environment default
when no runtime override exists. Additional projects remain available as
read-only projects.

Project names, colors, timestamps, timer and space data, the profile display
name, account defaults, and notification rules are restored from the file.
Importing account preferences and notification rules replaces matching
settings on the signed-in target account. The target account's email and
authentication identity do not change.

<Warning>
  Authentication sessions, provider credentials, webhook signing secrets,
  API-key tokens, and push-subscription credentials are bound to their source
  deployment and are not transferable. Recreate API keys and webhooks and
  enable browser push again on the target instance. Inbox history remains in
  the export as an archive but is not re-created on the target.
</Warning>

See [Self-hosting](/guides/self-hosting) for deployment setup and limit
configuration.

## Conflicts

Project identity is matched by project id.

- The Settings uploader uses the safe `skip` strategy. Existing target projects
  are left unchanged and reported as conflicts.
- API clients may explicitly request `replace`. Replacement is allowed only
  when the matching target project already belongs to the signed-in account.
- An id owned by another account is never read or overwritten.

## API

Download the signed-in account export:

```bash
curl --fail --cookie cookies.txt \
  --output tickward-account.json \
  https://your-tickward.example/api/account/export
```

Import it with the safe conflict strategy:

```bash
jq '{conflictStrategy:"skip", export:.}' tickward-account.json | \
  curl --fail --cookie cookies.txt \
    --header 'Content-Type: application/json' \
    --data-binary @- \
    https://your-tickward.example/api/account/import
```

Both endpoints require a signed-in account and are rate limited. Import
responses list created, replaced, conflicting, and read-only project ids, plus
the number of account-preference and notification-rule records applied.
