tickward runs as a Next.js app with PostgreSQL for durable data and a
Redis-compatible REST endpoint for rate limiting.

## Docker Compose

```bash
git clone https://github.com/CorgiCorner/tickward.git tickward
cd tickward
cp .env.example .env
# Edit .env and set POSTGRES_PASSWORD, DATABASE_URL, and DIRECT_URL.
docker compose pull
docker compose up -d
```

Open `http://localhost:3000`.

Generate a unique local database password with `openssl rand -hex 32`. Put it
in `POSTGRES_PASSWORD`, then use the same URL-safe value in `DATABASE_URL` and
`DIRECT_URL`. The example leaves all three values empty deliberately so a
committed development password cannot be reused in a real deployment.

The Compose stack starts the published tickward app image, PostgreSQL, Redis,
and an Upstash-compatible HTTP proxy for local rate limiting.

Published app images:

| Registry   | Image                                      |
| ---------- | ------------------------------------------ |
| Docker Hub | `docker.io/corgicorner/tickward:0.14.1`    |
| GHCR       | `ghcr.io/corgicorner/tickward:0.14.1`      |

Docker Compose reads `TICKWARD_IMAGE` from `.env`. The default example uses
Docker Hub. To use GHCR instead:

```bash
TICKWARD_IMAGE=ghcr.io/corgicorner/tickward:0.14.1 docker compose up -d
```

To build from a local checkout instead of using a published image:

```bash
docker compose up --build
```

## Local development

Use Node 22.

```bash
cp .env.example .env
npm ci
npm run dev
```

For development outside Docker, run PostgreSQL, apply migrations, and run a
Redis-compatible REST endpoint:

```bash
npm run db:migrate:deploy
```

Use host-reachable URLs in `.env` when you run services outside Compose.

For Web Push, generate a VAPID key pair with `npx web-push generate-vapid-keys`
(from the [`web-push`](https://github.com/web-push-libs/web-push) library; no
install needed, `npx` fetches it). It prints a **Public Key** and a **Private
Key**. Copy them into `WEB_PUSH_VAPID_PUBLIC_KEY` and
`WEB_PUSH_VAPID_PRIVATE_KEY`. Set `WEB_PUSH_VAPID_SUBJECT` to a contact
`mailto:` address such as `mailto:admin@example.com`, or to your site URL. It
identifies the sender to push services as required by the
[VAPID spec (RFC 8292)](https://datatracker.ietf.org/doc/html/rfc8292). Keep the
private key secret; the public key is exposed to browsers via
`/api/push/public-key`.

`SITE_URL` and `NEXT_PUBLIC_*` values affect build-time metadata or browser
bundles. Rebuild the app after changing them. The four plan-limit variables are
fallback defaults for the anonymous plan. Free account defaults double those
values, capped at 1000, and saved plan limits from the admin area override both
plans at runtime.

Webhook delivery needs a scheduler caller in production. See
[Webhooks and scheduler](/guides/webhooks) for the Cloudflare Cron, EventBridge,
and VPS cron options.

Set `TICKWARD_OWNERLESS_PROJECT_RETENTION_DAYS` to a positive day count when
you want scheduler ticks to remove anonymous or orphaned projects after that
retention period. Leave it unset or set it to `0` to keep cleanup disabled.

## Webhook signing secret encryption

Set `TICKWARD_ENCRYPTION_KEY` to a base64-encoded 32-byte key to encrypt webhook
endpoint signing secrets at rest. Generate one with:

```bash
openssl rand -base64 32
```

This setting is optional. When it is omitted, tickward keeps storing webhook
signing secrets as plaintext. Existing plaintext rows are upgraded
automatically the next time they are read.

## Accounts and email

Accounts are optional. Without account configuration, tickward still works with
anonymous projects and restore keys.

Restore keys are project access tokens for anonymous projects. Treat them like
passwords. When a signed-in user claims a project, tickward saves it to the
account and revokes the old restore-key access.

Timer share links are separate read-only public links. They expose the linked
timer view, not the full project or its restore key.

Email-code (OTP) sign-in is the only way to create or access an account, and it
requires **both** Better Auth and a configured email provider (Resend). Set all
of these together:

```bash
BETTER_AUTH_URL=http://localhost:3000
BETTER_AUTH_SECRET=change-me
RESEND_API_KEY=placeholder
RESEND_FROM="tickward <no-reply@example.com>"
RESEND_REPLY_TO=contact@example.com
```

Both halves are mandatory for accounts:

- Without `BETTER_AUTH_URL` and `BETTER_AUTH_SECRET`, every `/api/auth/*` route
  responds `501 auth_not_configured` and no sign-in is possible.
- Without `RESEND_API_KEY` (and `RESEND_FROM`), Better Auth loads but the
  verification code can never be delivered. OTP requests fail with "Email
  sign-in is not configured", so **accounts cannot be created or accessed**.

In other words, a working email provider is required to have an account. With
neither configured (the default), tickward runs anonymous-only with projects
and restore keys, and every other feature still works. These are runtime
variables, so you can add them to an existing deployment without rebuilding the
image.

OTP send requests are rate limited with Redis: one request per minute per
normalized email and three requests per minute per IP.

To smoke-test the request path against a running app:

```bash
SMOKE_BASE_URL=http://localhost:3000 SMOKE_EMAIL_TO=you@example.com npm run smoke:email-otp
```

This smoke test verifies that the app accepts and dispatches an OTP request. It
does not read an inbox or prove mailbox delivery.

## Database migrations

Managed deployments must apply Prisma migrations before building code that
depends on new tables or columns:

```bash
npm run db:migrate:deploy
```

## First administrator

After migrations and account sign-in are configured, choose either onboarding
path for the first administrator:

- Set `SEED_ADMIN_EMAIL`, then run `npm run db:seed:admin` to create or promote
  that account.
- On a fresh deployment with no administrator, sign in and visit `/setup`. The
  first signed-in user claims the administrator role.

Visiting `/admin` before an administrator exists redirects to `/setup`. Once any
administrator exists, `/setup` redirects away and cannot be used to claim
another one. The `/admin` area then hosts the plan limits editor. It can change
every anonymous and free account limit at runtime, including the saved snapshot
cap, and changes apply immediately.

## Demo data

Run Storybook locally with:

```bash
npm run storybook
```

To load screenshot-friendly demo timers and spaces, start the app and run:

```bash
npm run demo:seed
```

For a PostgreSQL-backed bootstrap seed, configure `DATABASE_URL` and
`SEED_ADMIN_EMAIL`, apply migrations, then run:

```bash
npm run db:migrate:deploy
npm run db:seed
```

The database seed creates or promotes the admin account and idempotently upserts
the demo project in PostgreSQL.

## Advanced database roles

Most installs can skip this.

If migrations and app runtime use different PostgreSQL roles, run this after
migrations:

```bash
npm run db:grant-runtime
```

Set `TICKWARD_DATABASE_RUNTIME_ROLE` or `TICKWARD_DATABASE_SCHEMA` only when the
script cannot infer them from `DATABASE_URL`.

## Docs

The Mintlify entrypoint is the root `docs.json` file on the public repo's
`main` branch. The `docs/site` directory mirrors the same content for app-local
checks and packaged docs assets.

If you host docs outside the app, set `TICKWARD_DOCS_ORIGIN` to the external
docs origin and the app will redirect `/docs` there.

Useful local docs checks:

```bash
npx mintlify@latest validate
npx mintlify@latest broken-links
```
